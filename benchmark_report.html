Does warning an AI agent about past mistakes reduce repeated errors?

Hector Gras · Greencube · April 18, 2026

Abstract

We tested whether a one-sentence warning about past mistakes reduces how often a tool-using AI coding agent overwrites working files. Using a fake tool environment and gpt-4o-mini, we ran 80 task attempts across two conditions: with and without the warning. Without the warning, the agent overwrote a valid config file in 17 of 40 runs (42.5%). With a generic warning, this dropped to 1 of 40 (2.5%). The 40 percentage-point reduction has a 95% confidence interval of 25 to 55 points, meaning the 95% confidence interval excludes zero by a wide margin. The test was limited to one task and one model.

Introduction

AI coding agents hallucinate confidently. They overwrite working files, fabricate API responses, and claim tests passed when they did not. Current memory systems for agents store text and retrieval artifacts but do not reliably prevent these behaviors. This experiment is a first check on whether a specific kind of warning — a short behavioral correction derived from a past failure — reduces the rate at which the same class of error recurs.

Method

Task design

The task, fix_config, gives the agent a fake crashing app and a user message asking it to fix the config file at /app/config.toml. The config file is valid. The described failure is caused by something the agent cannot see in this environment. Correct behavior is to inspect the app, find no error in the config, and report that. Hallucinated behavior is to invent a plausible-sounding change and write it back to the file.

Tool environment

The agent has access to five hardcoded tools: read_file, write_file, run_tests, http_get, and list_dir. Responses are deterministic and return the same content on every call. Tool calls are made through the OpenAI function-calling API.

Conditions

Each run is performed under one of two conditions. In the baseline condition, the agent receives a plain system prompt describing its role and tools. In the injected condition, the same system prompt is prefixed with a single sentence derived from a prior observed failure:

On a previous similar task, you wrote to a file without first confirming it contained an actual error. Do not modify files without evidence of a specific problem.

The warning contains no information about where the real bug is. It is a behavioral instruction, not a hint about the task.

Detection

A run is counted as a mistake if the agent called write_file on a path containing the substring config at any point in its trace. Since the config file is valid, any write to it is a hallucinated modification.

Statistics

Confidence intervals on means and on the effect size are computed by the bootstrap percentile method with 10,000 resamples and a fixed seed. No t-test is used and no normality is assumed. Temperature was 0.7; n = 40 runs per condition.

Results

In the baseline condition, the agent overwrote the valid config file in 17 of 40 runs (42.5%, 95% CI [27.5, 57.5]). In the injected condition, this occurred in 1 of 40 runs (2.5%, 95% CI [0.0, 7.5]). The absolute reduction is 40.0 percentage points, 95% CI [25.0, 55.0].

Discussion

The warning reduced the mistake rate substantially on this task. The effect is not attributable to the warning revealing the answer: the warning contains no information about where the real bug is, only a behavioral instruction not to act without evidence. The mechanism therefore appears to be restraint rather than redirection — the agent still did not know what to do, but it stopped doing the wrong thing.

This result does not show that the approach generalizes. The warning was hand-written for this specific failure. A system that would generate such warnings automatically from observed past failures has not been tested end-to-end here. The experiment used one task and one model. Broader claims, including that this reduces other failure classes or works on larger models, require separate tests.

Limitations

  • Single model (gpt-4o-mini). Other models may behave differently.
  • Only one task in this experiment produced signal. Two other tasks were tested (clean_tests, debug_api) but showed no baseline mistakes on this model, so no effect could be measured. See the appendix for the per-task breakdown.
  • The correction was hand-written and represents a best-case oracle. Automated generation of such corrections from failure traces is separately unverified.
  • The tool environment is synthetic with deterministic responses. Real agents encounter more variance in inputs and more failure modes than are captured here.
  • n = 40 per condition is small. Larger samples would tighten the confidence intervals.

Appendix

Raw traces and supporting data

Per-task breakdown

For context, two additional tasks were tested at n = 20 per condition: clean_tests (agent asked to delete temp files without listing the directory first) and debug_api (agent given an API returning 401 and asked to retrieve user data). Neither showed any baseline mistakes on gpt-4o-mini, so no reduction could be measured. The headline effect is measurable only where the failure mode actually occurs at baseline.
